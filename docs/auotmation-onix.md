## Overview

Automation Workbench ONIX acts as the primary integration layer for external Workbench interactions.

## Responsibilities

1. Serve as the point of contact for all external Workbench interactions:
    - Receive requests from buyer and seller NPs.
    - Forward requests to buyer and seller NPs from mock servers.
2. Handle all ACK/NACK conditions for ONDC protocol requests, including:
    - Schema validation
    - L1 validation (x-validation)
    - Context validation
    - Header and signature validation
3. Sign requests generated by mock servers.
4. Trigger Recorder Service calls to persist payloads in DB and transaction cache.

## Setup

### Run via Docker

1. Clone the generator repository:

    ```bash
    git clone https://github.com/ONDC-Official/automation-api-service-generator.git
    ```

2. Copy the relevant `build.yaml` to:

    ```
    src/config/build.yaml
    ```

3. Move into the repository:

    ```bash
    cd automation-api-service-generator
    ```

4. Install dependencies:

    ```bash
    npm i
    ```

5. Create `.env` from `.env.example` and provide required values.

6. Start development build generation:

    ```bash
    npm run dev
    ```

7. Start containers:

    ```bash
    cd build-output
    docker compose up --build
    ```

### Active Development Setup

1. Clone ONIX:

    ```bash
    git clone https://github.com/ONDC-Official/automation-beckn-onix
    ```

2. Update ONIX adapter configuration in:

    ```
    config/ondc-workbench-onix/adapter.yaml
    ```

    Set domain, version, and any development-specific configuration.

3. Generate payload schemas from the root of `automation-beckn-onix`:

    ```bash
    npx ondc-code-generator schema -c ./<path-to-build.yaml> -o ./schemas/ondc_ret10/v1.2.0 -f json
    ```

    Update domain/version values in the command as needed.

4. Clone plugins repository:

    ```bash
    git clone https://github.com/ONDC-Official/automation-beckn-plugins
    ```

5. Create `build.yaml` at the root of `automation-beckn-plugins`.

6. Generate L1 validations from the root of `automation-beckn-plugins`:

    ```bash
    npx ondc-code-generator xval -c ./build.yaml -o ./ondc-validator/ -l go
    ```

7. In `automation-beckn-onix/go.mod`, replace:

    ```
    replace github.com/beckn-one/beckn-onix => github.com/ONDC-Official/automation-beckn-onix v1.5.0
    ```

    with:

    ```
    replace github.com/beckn-one/beckn-onix => ./
    ```

8. In `automation-beckn-plugins/go.mod`, replace:

    ```
    replace github.com/beckn-one/beckn-onix => github.com/ONDC-Official/automation-beckn-onix v1.5.0
    ```

    with:

    ```
    replace github.com/beckn-one/beckn-onix => <ABSOLUTE_PATH_TO_AUTOMATION_BECKN_ONIX>
    ```

9. Create `.env` in `automation-beckn-onix` from `.env.example`.
   No `.env` is required for `automation-beckn-plugins`.

10. Build ONIX server from `automation-beckn-onix`:

    ```bash
    go build -o server cmd/adapter/main.go
    ```

11. Build plugins from `automation-beckn-plugins`:

    ```bash
    ./buildplugins.sh
    ```

12. Run ONIX server:

    ```bash
    ./server --config=config/ondc-workbench-onix/adapter.yaml
    ```

> Important: Re-run ONIX build (Step 10) after ONIX code changes and plugin build (Step 11) after plugin code changes.

#### Recorder Service Setup

If you use local cache or local DB, set up Recorder Service locally as well.

Refer to:
https://github.com/ONDC-Official/automation-recorder-service

After setup, update `mock-router.yaml` and `np-router.yaml` inside:

`automation-beckn-onix/config/ondc-workbench-onix`

to point to your local Recorder Service.

## Development Workflow

1. After ONIX server code changes, rebuild ONIX:

    ```bash
    go build -o server cmd/adapter/main.go
    ```

2. After plugin code changes, rebuild plugins:

    ```bash
    ./buildplugins.sh
    ```

3. Ensure Go package versions are aligned between ONIX and plugins.
   Mismatched versions can cause plugin loading failures in ONIX.

## Deployment

1. Base ONIX image:

    `ghcr.io/ondc/automation-onix:latest`

    This image is used as the base image in the final API service Dockerfile across domains and versions.

2. The `deploy-onix` workflow in config-store:
    - Clones `https://github.com/ONDC-Official/automation-api-service-generator`
    - Builds the final image in `build-output`
    - Pushes the image to `ghcr.io/ondc/automation-onix:latest` with relevant tags
    - The final deployment environment (EC2) then pulls this image

## Debugging

1. ONIX logs are available in Dozzle and can be filtered by container `ondc-onix`:

    http://13.233.69.163:8081/

2. Check cache entries in back-office:

    http://dev-automation.ondc.org/back-office

    Use key format:

    ```
    {transaction_id}::{subscriber_url}
    ```

3. Reproduce issues in Playground using NP payloads.

4. Reproduce locally and debug using logs and breakpoints.

    ### TIPS TO KEEP IN MIND

    - Always verify the **context** of the subject payload is correct and only proceed with further debugging if the context is correct.
    - before running the flows make sure /test endpoint is working fine by sending a sample payload. 

## FAQs

Use this section as a quick placeholder for common operational questions. Replace each placeholder answer as needed.

1. **Where can I view logs and inspect cache for troubleshooting?**
    - ONIX logs are available in Dozzle 
    - in ec2 docker logs can be viewed using `docker logs <container_id>`

2. **I am getting {action} not supported after {action} nack message** 
    - This is config loaded from config-service (supported actions) on start up only once and is not updated or reloaded until the server is restarted. Please check if the config in config-service is correct and restart the server to load the latest config.
    - On Startup this config is printed in logs with log message. Please check logs for the same and verify if the config is correct.

3. **Deployment is failing in github actions**
    - make sure bulid.yaml has correct domain and version in info section
    - make sure the docker image pushed to ghcr.io is public
    - make sure the env variables in github actions are correct
